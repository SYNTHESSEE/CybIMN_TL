## 🧠 Эксклюзивная интеграция: Архитектура QD³

В рамках данного проекта (Модуль Б: Логирование) мной была адаптирована и внедрена часть моей собственной концепции из академической курсовой работы — архитектура **QD³** (математическая дискретизация и коллапс энтропии).

**Суть QD³ в рамках курсовой:** 
Концепция QD³ изначально разрабатывалась для подавления стохастической неопределенности в Больших Языковых Моделях (LLM). Она базируется на полном лишении нейросети "свободы генеративной воли": любой входящий сигнал математически проецируется на жесткую многомерную матрицу (Блок-Вселенную аксиом), исключая семантические галлюцинации и переводя систему в детерминированное состояние.

**Применение QD³ в Кибериммунном Светофоре (Класс `LgoLog`):**
Я перенес этот же самый математический принцип в модуль логирования светофора для обеспечения абсолютной защиты от атак класса *Log Injection (внедрение в журналы)*. 

Как это работает здесь:
1. **Дискретизация сигнала (Квантование входа):** При обнаружении атаки, любой "мусорный" или вредоносный *payload* (например, `event.parameters`) не записывается в системный лог "как есть". Он воспринимается как чистый шум.
2. **Проекция на матрицу:** Потенциально опасная строка прогоняется через функцию `psi_function`, где она "коллапсирует" (сводится) в безопасную, строго детерминированную целочисленную константу (например, код ошибки `104`).
3. **Безопасный рендер:** Монитор обращается к локальной изолированной Блок-Вселенной (словарю `_err_map`) по полученному коду и выдает в консоль жестко захардкоженный, эталонный текст (`"CRITICAL: INVALID_STATE_INJECTION"`).

Эта портированная из моей курсовой работы механика работает здесь ровно так же, как и там: **мы математически изолируем уязвимый выход (логер) от опасного, энтропийного входа (запрос атакующего)**. Хакер не сможет выполнить свой код через логирование, так как сигнал проходит полную дискретизацию.

## 🏗 Архитектура и реализованные модули

Система разделена на изолированные потоки, общение между которыми разрешено только через брокера — `Monitor`.

* **Модуль А (Инвариант состояний):** Внедрена статичная карта безопасных свечений `ALLOWED_STATES`. Компоненты (`ControlSystem`, `LightsGPIO`) переведены в режим автономной работы `while True`, реализуя жизненный цикл промышленного контроллера.
* **Модуль Б (Монитор безопасности):** Интеграция Zero-Trust подхода. Любой `Event` проходит сквозь валидацию структуры, источника и ограничений на частоту вызовов.
* **Модуль В (Нейтрализация атак):** Система устойчива к:
  * *Инъекции состояний (CybTL_01)*: Проверка на строгое соответствие whitelist.
  * *Подмена источника / Спуфинг (CybTL_02)*: Проверка регистрации отправителя.
  * *DoS и Flood атаки (CybTL_03)*: Индивидуальный Rate-Limiting по ключу отправителя (`defaultdict`). Изолирует атакуемого без нарушения работы соседних модулей.
  * *Replay-атаки (CybTL_04)*: Сверка `event.timestamp` (допуск отклонений $\le$ 1 секунда).
* **Модуль Г (Интеграция с внешней системой):** Подключен мок-сервер городской инфраструктуры `CitySystemConnector` со строгой двусторонней авторизацией и требованием проставления флага доверия `from_city=True`.

Проект автономен, использует встроенные модули Python. Для прохождения автоматических проверок потребуется пакет `pytest`.

**Установка:**
```bash
python -m pip install pytest
```
